00 The practice we are supporting

How P&IDs are produced

A P&ID is not drawn in one pass. It is the end of a chain that starts coarse and adds detail in deliberate layers — which is exactly the “very basic block levels first, then progressively more” approach you described. Understanding that chain tells us what the tool must let a user do at each stage.

STAGE 1
Block Flow

Blocks for unit operations, arrows for flow. The process concept, no equipment detail. Conceptual / pre-FEED.

STAGE 2
Process Flow

Major equipment, main streams, key conditions and a material balance. Answers “what equipment, what duty”.

STAGE 3
P&ID

Every line with number, size and class; all valves, instruments, control loops, safety devices. The build-and-operate document.

STAGE 4
Lifecycle

HAZOP, construction, commissioning, operation, MOC and as-built revision — the P&ID is the living reference throughout.

What a finished P&ID carries

Equipment shown as schematic symbols sized to host their nozzles; piping with line numbers encoding service and conditions, plus size and pipe class; valves and special piping items; instruments and control loops in ISA bubbles; off-page connectors where lines leave the sheet; and a frame of context — title block, revision table, reference drawings, a legend/symbol sheet, and engineering notes. Importantly, a P&ID is schematic: valves are drawn at a single nominal size, and a valve’s size is annotated only when it differs from the line it sits in.

The decisive distinction: drafting vs. data

The single most important lesson from the established tools is the split between drawing-centric CAD (lines and text that merely look right) and data-centric intelligent P&ID (every placed item is an object linked to an engineering record). In the data-centric model, properties entered against an object drive its on-drawing labels and feed reports directly, connectivity between items is real and queryable, and the same database backs automatic validation. This is the model worth building toward — and it is the same “data is the source of truth, the graphic is a projection of it” principle established in the standards reference.

▸ Why this matters for us

If labels are free text and lines are just geometry, the tool can never generate a line list or catch a duplicate tag. Every requirement below assumes the object-with-record model. The graphic is rendered from the data, never the reverse.

01 Derived from the practice

Guiding principles

  • P1Data is the source of truth. Each component is a record; the drawing is a projection. Labels and reports read from the record.
  • P2Progressive elaboration. The same model supports a coarse block diagram and a fully detailed P&ID. Detail is added, not redrawn.
  • P3Reuse over redraw. Repeated arrangements are captured once as typicals and instantiated, not copied by hand.
  • P4Connectivity is real. A connection between two ports is a modelled relationship, not two lines that happen to touch.
  • P5Validate continuously. Integrity checks run as the user works and on demand, surfacing issues early.
  • P6Standards are pluggable. Symbol set and tagging scheme (ISA / KKS / RDS-PP) are configuration, not hard-coding.
  • P7Portable and self-contained. One HTML file, no install, no network dependency, works offline from the filesystem.

02 What the software must do

Functional areas

Eleven capability areas, A–K. Each lists its core requirements; the consolidated, prioritised list with phase mapping is in the requirements register.

A Drawing surface & sheet

  • A1A pannable, zoomable canvas with an optional snap grid.
  • A2A sheet frame with title block, border zones, revision table and a notes area.
  • A3Place, move, rotate, mirror and delete components; multi-select and align.

B Component model & symbol library

  • B1Each symbol is a registered component type: an SVG render function plus a declared field schema.
  • B2Placing a symbol creates an instance carrying a stable ID and its own data record.
  • B3A seed library covering the core classes: equipment, valves, piping items, instruments, lines.
  • B4New types added by registering a module — no change to the core engine.

C Connectivity & topology

  • C1Components expose ports (equipment nozzles, valve ends, instrument taps) as named connection points.
  • C2Lines connect port-to-port as modelled relationships; endpoints snap to ports.
  • C3Flow direction is held on the connection and shown by arrowheads.
  • C4Moving a component keeps its connections attached and re-routes lines.

D Identification & tagging

  • D1A pluggable tagging scheme — ISA loop tags, KKS, or RDS-PP — selected per project.
  • D2Auto-numbering with per-scheme validation and parsing of structure.
  • D3Project-wide tag uniqueness enforced; duplicates flagged.
  • D4On-drawing tag labels are driven by the record, never edited as free text.

E Line designation & process data

  • E1Lines carry a line number, nominal size, service and pipe class/spec.
  • E2Valves carry type, end connection and fail-action; size shown only when it differs from the line.
  • E3Instruments carry measured variable and function per ISA-5.1, grouped into control loops.

F Reusable typicals / assemblies

The headline capability — detailed in section 03. In brief: capture a sub-arrangement once, instantiate it many times, and let it adapt to what it connects to.

G Validation & integrity

  • G1Rules-driven checks: duplicate tags, missing mandatory data, dangling (unconnected) ports.
  • G2Connection-logic checks: incompatible port types, illegal connections, spec breaks at boundaries.
  • G3Checks run both live (as drawn) and as an on-demand audit producing a findings list.

H Off-page & multi-sheet

  • H1Off-page connectors terminate a line leaving the sheet and reference its continuation.
  • H2Matched connectors across sheets resolve to the same logical line.
  • H3A drawing register tracks sheets, numbers and revisions across the set.

I Reports & deliverables

  • I1Generate a line list, valve list, equipment list and instrument index from the model.
  • I2Reports reflect live data — no separate re-keying — and export to CSV.

J Persistence, revision & exchange

  • J1Save and load the whole model as JSON; round-trip is lossless.
  • J2Revision history with a per-revision description (MOC-style audit trail).
  • J3Export shaped for DEXPI / ISO 15926 mapping; import where feasible.

K Non-functional

  • K1Single self-contained HTML file; no external fonts, scripts or network calls.
  • K2Runs offline from the filesystem on a managed laptop; prints cleanly.
  • K3Remains responsive at the scale of a realistic single sheet (hundreds of objects).

03 Reuse that adapts

Typicals — the headline capability

This is the concept you flagged: snippets of an area that get reused and adjusted for what they connect to. It has a long pedigree in P&ID practice and a clear shape in the intelligent tools, so it is worth specifying carefully rather than discovering later.

◢ Grounded in practice

Where a valve or piping arrangement recurs — a control-valve station with isolation and bypass, a pump with its suction/discharge valves, a steam-trap set — practice is to draw it once as a typical detail on the symbols sheet and reference it, rather than redraw it each time. The tool should make that a first-class, live object.

What a typical is, in the model

A typical is a saved sub-graph of the same component model: a small set of components and the connections between them, with a defined boundary. It is not a picture — it is a parameterised fragment of the data model that renders like everything else.

  • F1Capture. Select an arrangement and save it as a named typical, recording its components, internal connections and exposed boundary ports.
  • F2Instantiate. Drop a typical onto the canvas as one act; it expands into its constituent objects, each with fresh stable IDs.
  • F3Parameters. A typical declares parameters (e.g. line size, service, valve type, tag seed) set at insertion, so one definition serves many cases.
  • F4Boundary ports. A typical exposes named inlet/outlet ports; these are how it connects to the rest of the drawing.
  • F5Context adaptation. On connection, boundary attributes adapt to what they join — a typical inserted into a 6"-CWS line inherits that size, service and spec unless explicitly overridden.
  • F6Connection rules. Ports declare compatibility; the tool prevents or flags joins that violate them (e.g. instrument signal to process line).
  • F7Independence after insertion. An instance is editable on its own; later edits to the definition do not silently rewrite placed instances (no surprise mutation).
▸ Design consequence

F5 and F6 are what make a typical “adjust for what it connects to”. They require two things already in the model: ports with declared attributes/compatibility (area C) and an attribute-inheritance rule at the boundary. Get ports right early and adaptive typicals become a natural extension rather than a rebuild.

04 Build order

Phased roadmap

Each phase is usable on its own and sets up the next. The progression mirrors how P&IDs themselves evolve — coarse blocks first, detail and intelligence layered on. The highlighted phase is where “reuse that adapts” arrives.

P0

Block level

Place labelled blocks, connect them with simple directional lines, save/load as JSON. Establishes the source-of-truth model and the canvas.

Delivers · model + canvas + persistence (P1, A1–A3, B1–B2, C2–C3, J1)
P1

Object & symbol detail

Real symbol library, component records with properties, a tagging scheme with uniqueness, property-driven labels.

Delivers · symbol registry + tagging (B3–B4, D1–D4, E1–E3)
P2

Ports, connectivity & validation

Ports/nozzles as named connection points, real port-to-port connections that survive moves, and the first integrity checks.

Delivers · topology + checks (C1, C4, G1–G3)
P3

Reusable, connection-aware typicals

Capture and instantiate typicals; parameters at insertion; boundary ports; attribute inheritance and connection rules so a typical adapts to what it joins. This is the snippet capability.

Delivers · typicals (F1–F7)
P4

Multi-sheet, reports & exchange

Off-page connectors and a drawing register; generated line/valve/equipment/instrument lists; revision history; DEXPI-shaped export.

Delivers · scale-out (H1–H3, I1–I2, J2–J3)

06 Feeding the next document

Implications for the schema

These requirements pin down what the component record must hold before any code is written — which is precisely what the schema spec (the agreed cycle-1 deliverable) defines:

  • S1Identity: a stable ID on every instance (B2, F2), plus a tag held separately from identity (D1–D4).
  • S2Class: a component type/class reference (B1) that can later map to a DEXPI / ISO 15926 class (J3).
  • S3Two attribute namespaces: geometry/visual (position, rotation) kept apart from domain data (tag, size, service, spec, ratings).
  • S4Ports: a named-port collection per component, each port carrying attributes and compatibility (C1, F4–F6).
  • S5Connections: first-class relationship records referencing two ports, carrying flow direction and line data (C2–C3, E1).
  • S6Typical definitions: a sub-graph record (components + internal connections + boundary ports + parameters) distinct from its instances (F1–F3).
  • S7Document frame: sheet, title-block, revision and notes records around the component model (A2, H3, J2).
▸ Hand-off

S1–S7 are the headings the schema spec should fill in. The functional work has done its job: it has told us exactly what the record must carry, so the schema is now a description task, not a guessing one.
