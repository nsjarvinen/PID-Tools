00 What this is

Intent & boundaries

This document fixes the data shape before any code is written. It is the cycle-1 deliverable agreed under the charter: a written schema spec, no implementation. Everything downstream — rendering, tagging, validation, typicals, reports — reads and writes this structure.

It answers exactly the seven headings the functional requirements identified (S1 identity, S2 class, S3 two namespaces, S4 ports, S5 connections, S6 typicals, S7 document frame), and nothing more. Behaviour lives in later cycles.

The one load-bearing rule

The model defined here is the single source of truth. The on-screen P&ID is a projection rendered from it. Geometry that drives the picture and engineering data that the database cares about live in the same record but in separate namespaces, so neither contaminates the other.

Code vs. data — a deliberate split

Two things are easy to confuse. Component types (the gate-valve definition: its schema, default ports, render function) live in the code-side registry and are not part of the saved file. Component instances (this particular valve, here, tagged thus) are data and are saved. A saved model references a type by typeId; it never embeds the type's render logic.

▸ Why the split matters

It keeps saved files small and portable, lets a type's symbol be improved without rewriting every saved drawing, and means the field schema a type declares is the future database column set for that class.

01 Orientation

The model at a glance

One root object holds everything. Components and connections are flat collections keyed by ID; the document frame and typicals sit alongside them.

PidModel ├── schemaVersion "0.1" ├── project name, tagging scheme, defaults ├── document title block, revisions[], notes[] (S7) ├── sheets[] one per drawing sheet (S7) ├── components[] instances — graphic + record (S1–S3) │ └── ports[] named connection points (S4) ├── connections[] port-to-port relationships (S5) └── typicalDefs[] reusable sub-graphs (S6)

Components do not nest inside one another and connections are not nested inside components. Both are top-level lists that reference each other by ID — the same flat, ID-keyed shape a database uses, which is what makes export a mapping job rather than a rewrite.

03 S1 · S2 · S3 · the core record

Component record

The heart of the model. Note the three separations: identity (a stable ID) is distinct from tag (an engineering label that can change); geometry (drives the render) is distinct from domain (the engineering data). Identity never changes; everything else can.

FieldTypeReqS-refNotes
idUUIDyesS1Stable from creation. The future primary key. Never reused.
typeIdstringyesS2Reference into the code-side registry, e.g. "valve.gate". Maps to a DEXPI/ISO 15926 class later.
sheetIdstringyesS7Which sheet the instance sits on.
geometryGeometryyesS3Visual namespace: position, rotation, mirror.
tagTagnoS1{ scheme, value }. Absent until tagged. Separate from id.
domainobjectnoS3Engineering namespace. Shape defined by the type's field schema.
portsPort[]noS4Instantiated from the type; may carry per-instance overrides.
typicalInstanceIdUUIDnoS6Present if this component was placed as part of a typical instance.
 
// Component — a gate valve on the cooling-water line
{
  "id": "c-9f2a…",
  "typeId": "valve.gate",
  "sheetId": "sh-024",
  "geometry": { "x": 420, "y": 300, "rotation": 0, "mirrored": false },
  "tag": { "scheme": "KKS", "value": "P1PCC10AA001" },
  "domain": { "valveType": "gate", "endConnection": "flanged",
              "failAction": null, "nominalSize": null },
  "ports": [ /* see §04 */ ]
}
▸ The two-namespace discipline

The renderer may read geometry and tag.value only. Reports and the future database read domain and tag. Nothing reads geometry as if it were engineering data, and no engineering datum is inferred from pixel position. This is what keeps the model exportable.

04 S4 · connection points

Ports

A port is a named place a connection may attach — an equipment nozzle, a valve end, an instrument tap. Ports carry the attributes that make a typical able to adapt to what it connects to, and the compatibility that lets validation reject illegal joins.

FieldTypeReqNotes
idstringyesUnique within its component, e.g. "in", "out", "N1".
namestringnoHuman label, e.g. "Suction".
kindenumyesprocess | signal | instrument | electrical.
offset{dx,dy}yesPosition relative to the component origin (visual).
compatibleKindsenum[]noWhich kinds it may join. Drives connection rules (F6, G2).
attributesobjectno{ nominalSize?, service?, pipeClass? } — the values a typical inherits at its boundary (F5).
 
// Two process ports on the gate valve
"ports": [
  { "id": "in",  "kind": "process", "offset": { "dx": -12, "dy": 0 },
    "compatibleKinds": ["process"], "attributes": { "nominalSize": null } },
  { "id": "out", "kind": "process", "offset": { "dx":  12, "dy": 0 },
    "compatibleKinds": ["process"], "attributes": { "nominalSize": null } }
]

05 S5 · the topology

Connections

A connection is a first-class relationship between two ports — not two lines that happen to touch. For process connections it also carries the line data (number, size, service, class) that a line list is generated from.

}

06 S6 · reuse that adapts

Typical definitions

A typical is a saved sub-graph — components, the connections between them, exposed boundary ports, and parameters — kept separate from the instances made from it. Placing one expands it into ordinary components and connections with fresh IDs, sharing a typicalInstanceId, with parameters applied and boundary attributes resolved against whatever it joins.

FieldTypeReqNotes
idstringyesDefinition identity, e.g. "typ.cv-station".
namestringyese.g. "Control valve station w/ bypass".
parametersParam[]no{ name, type, default }, set at insertion (F3).
componentsTemplateComp[]yesLike Component but with relative refs and param placeholders.
connectionsTemplateConn[]yesInternal topology, by local component ref.
boundaryPortsBoundaryPort[]yes{ name, componentRef, portId } — how the typical connects outward (F4).
▸ Definition ≠ instance

A placed typical is fully expanded into real components and connections — it is not a live link back to the definition. Editing a definition later does not rewrite placed instances (F7). The typicalInstanceId on each component only records shared provenance, so an instance can be selected, moved, or deleted as a unit.

07 It all fits together

Worked example

A minimal but complete model: one cooling-water pump discharging through one gate valve, on one sheet. This is the shape a Phase-0 save/load round-trip (J1) must reproduce losslessly.

▸ Round-trip is the Phase-0 test

Parse this JSON → build the in-memory model → serialise it back → the two JSON texts are equivalent. That single assertion is the first self-test (J1) and proves the source-of-truth model holds together before any pixel is drawn.

08 Calls made — and still open

Open decisions

Three choices are baked into the above and worth your explicit sign-off, because each has a defensible alternative.

◢ Decision 1 — a line is a connection, not an object

A pipe is modelled as a connection carrying line data, not as its own component. This keeps topology first-class and the model simple.

Trade-off: a single physical line running through several in-line items (valve, reducer, tee) becomes several connection segments. A later "line group" concept may be needed to treat them as one line for the line list. Acceptable for Phases 0–2; flag if you'd rather model a pipe segment as a component now.

◢ Decision 2 — the type registry is code, not data

Type definitions (schema, default ports, render) live in code; saved files reference them by typeId. Portable and small.

Trade-off: a saved file opened against a newer build could meet a typeId whose schema has changed. Mitigated by schemaVersion and migration (§09), but worth noting that files are not fully self-describing without the code.

◢ Decision 3 — IDs are opaque UUIDs, tags are separate

Identity is a UUID assigned at creation; the engineering tag is a separate, editable field. A component can exist untagged.

Trade-off: two identifiers to carry. The payoff is that re-tagging never breaks connections or references — which is the whole point of S1.

09 Forward compatibility

Versioning & DEXPI alignment

schemaVersion is checked on load. A loader for an older version runs a migration to the current shape before the model is used; migrations are additive where possible and never silently drop data. Bump the minor version for additive fields, the major for breaking changes.

The structure deliberately mirrors how DEXPI carries a P&ID, so a future export (J3) is a field-mapping exercise, not a redesign:

This schemaMaps toward (DEXPI / ISO 15926)
component.typeIdEquipment / Piping component class
component.domain.*GenericAttributes on the object
connection (+ line)PipingNetworkSegment / signal line, with topology
portNozzle / connection point
tagObject identifier / designation
▸ Not now

No DEXPI vocabulary is adopted at version 0.1 — only the shape is kept compatible. Committing field names to DEXPI classes is a later cycle, and one to do deliberately rather than by accident.
