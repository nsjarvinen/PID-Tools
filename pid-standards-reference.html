00 Orientation

Scope & the three-layer model

This reference collects the standards that matter when building a portable, single-file HTML tool for authoring process P&IDs for power generation plant. It is written against a real sample: the as-built P&ID set for Alinta Cogeneration (Pinjarra) Unit 1, a Downer / Mitsubishi consortium project — specifically the HP economisers sheet, the cooling-water sheet, and the symbol-and-legend sheet.

That set is instructive because it is a hybrid: ISA-style instrument bubbles, KKS-style equipment tags, and an ISO/DIN-flavoured general symbol set, all on the one drawing. A tool serving this asset base has to speak all three.

The standards sort into three layers, and they map cleanly onto three modules the programme needs anyway: what the shapes mean, how things are named, and how the data travels.

01 What the shapes mean

Symbols — the render library

The canonical symbol-and-identification standard — and the one the legend sheet is built on — is ANSI/ISA-5.1. It was revised and ANSI-approved in mid-2024, with the title changing to “Instrumentation and Control — Symbols and Identification” to acknowledge that modern work is about integrated control functions rather than individual instruments. First published in 1949, it remains the most widely used standard of its kind. The 2024 edition is now accompanied by two technical reports — ISA-TR5.1.02 (identification guidelines) and ISA-TR5.1.03 (graphic symbol guidelines) — which carry much of the explanatory material that used to live in the annexes.

The instrument letter-code system is the heart of it: a first letter for the measured or initiating variable (A analysis, F flow, L level, P pressure, T temperature, and so on) followed by succeeding letters for the readout or output function (I indicate, T transmit, C control, etc.), then a loop number. FIC045 is the flow indicating controller in loop 045; FT045 is the flow transmitter in the same loop.

◢ On the Pinjarra set

The letter-code table on the symbol-and-legend sheet (XP-101) is exactly this ISA-5.1 identification system. The instrument bubbles throughout the HP economiser and cooling-water sheets follow its function/location conventions.

The non-instrument graphics

For valves, equipment, line types, and fittings, the international references are ISO 14617 (graphical symbols for diagrams — a multi-part set, with part 6 covering measurement and control symbols that align with ISA) and ISO 10628 (flow diagrams for process plants). Underlying many of these is IEC 60617, a very large graphical-symbol database, while ISO 15519 specifies the diagram conventions for the process industry. Regionally, DIN 28004 and BS 5070 still appear on European-consortium jobs — relevant here, since a Mitsubishi/Downer house style is likely DIN-influenced even where it leans on ISA.

▸ Build consequence

Don’t hard-code symbols as fixed SVG. Each symbol is a registered component type — an SVG render function plus a field schema — so a new symbol is one self-contained module that registers itself. The standards above simply give you the canonical set to seed that library.

02 How things are named

Identification — the tagging engine

This is where the sample drawings are genuinely hybrid, and it forces a real design decision. The instrument bubbles use ISA-5.1 loop-style tags, but the equipment is tagged in KKS — the Kraftwerk-Kennzeichensystem, or power-station identification system. Tags such as P1HAC34AC110, P1PCD05AC005, and P1PCC10AP010 are KKS designations: a block/unit prefix, a function (system) code, an equipment-unit code, and a component code.

The current English edition is the VGB 8th revised edition (2018). KKS is being superseded by RDS-PP (Reference Designation System for Power Plants), which is built on the ISO/IEC 81346 structuring-and-reference-designation series — specifically 81346-10 for power plants (formerly ISO/TS 16952-10). Within that series, 81346-1 gives the basic rules and 81346-2 the classification letter codes for object classes. RDS-PP also draws in companion standards for signals (IEC 81175), connections, and document classification (IEC 61355).

◢ On the Pinjarra set — and why it matters

Existing KKS-coded plants are not recoded to RDS-PP; KKS continues to be applied for additions, conversions, and I&C retrofits. A tool serving an asset like Pinjarra must therefore speak KKS natively, not only its successor.

▸ Build consequence

Tagging must be a pluggable scheme, not baked in. A chemical plant wants ISA loop numbers; a power station wants KKS or RDS-PP. Make the identifier a strategy module the user selects, each carrying its own validation and parsing rules. The research has just handed you this module boundary.

03 How the data travels

Data exchange — the database layer

This is the layer that speaks directly to the goal of attaching database-ready data to each model component, and it has matured at the right moment. The relevant standard is DEXPI (Data Exchange in the Process Industry): a vendor-neutral, open data model for P&IDs that functions as a pragmatic implementation of ISO 15926, focused on high-fidelity transfer of graphics, topology, and metadata. It is serialised through the Proteus XML schema. DEXPI Specification 2.0 was released in late 2025 with a unified XML serialisation format.

DEXPI models precisely what you would want hanging off each component: equipment data, pipeline and segment data with plant-wide unique numbering, instrumentation, and the topology — the logical connectivity — between objects. On the instrumentation side it reconciles ISA-5.1 with IEC 62424, the standard for representing process-control requests in P&IDs and exchanging them with control-engineering tools. (IEC 62424’s CAEX data format is also the ancestor of AutomationML, IEC 62714.)

▸ Build consequence

You almost certainly don’t want to implement full Proteus XML now — but let DEXPI shape the data model so a future export is a mapping job, not a re-architecture. Each component carries a stable ID, a class that can map to a DEXPI / ISO 15926 class, explicit topology references (this nozzle connects to that segment), and a generic-attributes bag for the domain data. The graphic is a projection of that record, never the source of truth.

05 From standards to architecture

What this means for the build

The research hands us three clean module boundaries, with one underlying principle holding them together: the data is the source of truth and the graphic is a projection of it. If the SVG holds the state, you can never cleanly export; if a plain data model holds the state and the graphic renders from it, export becomes serialisation rather than a retrofit.

Symbol / render registry

Each symbol is a registered component type — an SVG render function plus a declared field schema. Adding a type touches nothing else. The field schema a component declares is its future database table.

Pluggable tagging engine

Identification is a strategy module the user selects — ISA loop numbers, KKS, or RDS-PP — each with its own parsing and validation. The tool must read and write KKS to serve existing power-station assets.

DEXPI-shaped data model

Every component carries a stable ID, a class, explicit topology references, and a generic-attributes bag — geometry/visual attributes kept in a separate namespace from domain attributes. Committing to this shape now keeps every database route open: export, embedded, or server-backed later.

The single decision worth making deliberately up front is the third. A DEXPI-flavoured record shape costs almost nothing now and is expensive to retrofit — which is exactly why the export-vs-embedded-vs-server database question can wait.
